## Release Checklist

Run through this list every time you cut a tag. Treat it as a hard gate, not a guideline.

### 1. Pre-flight (on the release branch)

- [ ] CI green on the head of `main` (or the LTS branch you are releasing from).
- [ ] No PRs marked `release-blocker` open.
- [ ] `PublicAPI.Unshipped.txt` reviewed — every line is intentional.
- [ ] CHANGELOG draft generated by Release Drafter reviewed end-to-end.
- [ ] Sample scenes open and play without errors on Unity 2022 LTS AND Unity 6.
- [ ] Benchmark suite ran locally with no >5% regression versus last release.

### 2. Version bump

- [ ] Decide MAJOR / MINOR / PATCH per [SEMVER_POLICY](SEMVER_POLICY.md).
- [ ] Update `version` in every `package.json` under `Assets/RevCore/*` to the new version.
- [ ] Update `version` in the root README install snippet.
- [ ] If pre-1.0 MINOR with breaking changes: ensure migration note exists in CHANGELOG.

### 3. PublicAPI promotion

- [ ] Move every entry from `PublicAPI.Unshipped.txt` to `PublicAPI.Shipped.txt` (per module).
- [ ] Sort `PublicAPI.Shipped.txt` (analyzer expects sorted).
- [ ] `PublicAPI.Unshipped.txt` reset to the file's required header.

### 4. CHANGELOG finalize

- [ ] Replace `## [Unreleased]` with `## [X.Y.Z] - YYYY-MM-DD`.
- [ ] Add a new empty `## [Unreleased]` header at the top.
- [ ] If breaking changes: include a "## Migration from X.(Y-1)" section with before/after.

### 5. Commit & tag

- [ ] Commit titled `chore(release): vX.Y.Z`.
- [ ] Tag annotated: `git tag -a vX.Y.Z -m "vX.Y.Z"`. Never lightweight tags.
- [ ] Push branch and tag: `git push origin main && git push origin vX.Y.Z`.

### 6. GitHub Release

- [ ] `release.yml` workflow creates the draft. Edit it to attach the CHANGELOG section as the body.
- [ ] Mark as pre-release if version contains `-rc` / `-beta` / `-alpha`.
- [ ] Publish.

### 7. Post-release

- [ ] Verify install via UPM URL with the new tag works from a fresh empty Unity project (one of the consumer projects).
- [ ] Announce in the dev channel with link to release notes.
- [ ] If LTS-relevant: create or update `v<MAJOR>.<MINOR>-lts` branch from the tag.

### 8. Rollback (if needed within 48h)

- [ ] Delete the GitHub Release (or mark draft).
- [ ] Do NOT delete the git tag — consumers may have pulled it. Cut a new patch instead.
- [ ] File an incident note under `docs/incidents/` describing what slipped through the checklist.
